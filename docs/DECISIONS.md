# Decisions (ADR-lite)

Append-only. Format: context / choice / why / alternatives. Newest last.

---

## D-001 · 2026-07-16 · Phase 1 covers all three feed formats

- **Context:** the plan lists semt.002 + MT535 + camt.053 for Phase 1; starting with fewer parsers was considered as a smaller first slice.
- **Choice:** build all three in Phase 1.
- **Why:** the point of the phase is the multi-format story — the *same* portfolio expressed in three wire formats, parsed to one common model; doing it in one pass keeps that comparison honest.
- **Alternatives:** semt.002 + MT535 first with camt.053 as a fast follow; semt.002 alone.

## D-002 · 2026-07-16 · Seed portfolios from real SEC 13F filings

- **Context:** synthetic positions can be random tickers or modelled on real institutional books.
- **Choice:** seed from 13F-HR filings of 2–3 well-known filers.
- **Why:** positions look like a real book rather than noise; the EDGAR pull doubles as proof of the scheduled-fetch path; public-domain data, small volume.
- **Alternatives:** fully synthetic (simpler, looks artificial); defer 13F until the fetch path exists.

## D-003 · 2026-07-16 · Quarkus + jOOQ for the serving layer

- **Context:** the serving stack could be Quarkus or Spring Boot; either works with jOOQ.
- **Choice:** Quarkus + jOOQ.
- **Why:** fast startup and low memory in containers (with native image as a further option), which directly cuts hosting cost; jOOQ keeps the SQL explicit and reviewable instead of hidden behind an ORM — in a data platform, the queries *are* the product.
- **Alternatives:** Spring Boot + jOOQ (larger ecosystem, heavier footprint).

## D-004 · 2026-07-16 · Reference data = small real slice

- **Context:** the securities master could be invented, a small real slice, or a bulk pull.
- **Choice:** real data for ~50–100 securities: OpenFIGI mappings + SEC ticker/CIK + a handful of GLEIF LEIs.
- **Why:** real identifier-mapping pain at trivial volume, licence-safe; a fully synthetic master would mean reconciling against invented data.
- **Alternatives:** fully synthetic; full GLEIF golden copy (more scale, little extra value here).

## D-005 · 2026-07-16 · Serving runs on real AWS via Terraform; PaaS dropped

- **Context:** the original plan hosted the live site on a PaaS (Railway) with Terraform merely *describing* an AWS production target that would never be applied. Expected usage is tiny (a single owner plus occasional demos), so cost is not the binding constraint.
- **Choice:** RDS Postgres `db.t4g.micro` + AWS App Runner for the Quarkus container, frontend on Vercel free, all provisioned by Terraform. Deploy path: GitHub Actions → ECR → App Runner.
- **Why:** infrastructure-as-code that is never applied is fiction; running the real thing keeps the Terraform honest, uses one cloud instead of two, and App Runner covers the PaaS conveniences (public TLS endpoint, no load balancer to manage).
- **Alternatives:** Railway as planned (simplest ops, Terraform stays unapplied); hybrid PaaS + apply/destroy AWS demo cycles (two platforms to maintain).
- **Guardrails:** AWS budget alert before the first resource; no NAT gateway, no ALB, no always-on Aurora Serverless — each is a fixed monthly cost the workload doesn't justify. Verify App Runner availability on the account's plan before writing Terraform; fall back to ECS Fargate.

## D-006 · 2026-07-16 · External data fetching runs on GitHub Actions

- **Context:** Databricks Free Edition serverless compute has a restricted outbound-internet allowlist, so pipeline jobs cannot call external sources (SEC EDGAR, price APIs) directly. Something with open egress must fetch data and land it where Databricks can read it.
- **Choice:** GitHub Actions cron workflows on GitHub-hosted runners in this repo, pushing raw files into a Unity Catalog volume via the Databricks CLI/REST API. Databricks Workflows still schedules all lakehouse processing.
- **Why:** zero standing infrastructure and zero cost (free minutes for public repos); cron, secrets management, and run logs are built in; the fetch code lives beside the rest of the codebase and works from day 1 — before any AWS infra exists.
- **Alternatives:** fetching from Databricks Workflows directly (blocked by the egress restriction); AWS EventBridge Scheduler + Lambda (the cloud-native equivalent — viable, but requires AWS infra and Terraform earlier than needed; a reasonable later migration); an always-on personal machine (unreliable).

## D-007 · 2026-07-16 · Environments: local dev + CI + one live environment; no standing staging

- **Context:** production platforms typically run dev/staging/prod. This is a solo-maintained reference build with demo-scale traffic and no users at risk.
- **Choice:** three layers — local Docker dev (Postgres pinned to the RDS major version), CI on every PR (lint + tests, required before merge), and a single live AWS environment.
- **Why:** a standing staging stack would roughly double infra cost and maintenance for no risk reduction that CI + local parity doesn't already provide at this scale.
- **Alternatives:** permanent staging (cost/upkeep unjustified); ephemeral staging via Terraform workspaces — documented as the scale-up path: the same modules can stand up an identical stack briefly, then destroy it.

## D-008 · 2026-07-16 · Python toolchain: uv + ruff + pytest + Pydantic, pinned to 3.12

- **Context:** the first Python code (ingest/) needs environment management, linting, and testing that behave identically on a dev machine and in CI; the canonical model carries untrusted feed data.
- **Choice:** `uv` for environments + lockfile, `ruff` for lint + format, `pytest` for tests, Pydantic v2 for models. Python pinned to 3.12 via `.python-version` to stay close to Databricks serverless runtimes.
- **Why:** uv is a single fast binary with lockfile-first discipline; ruff replaces three legacy tools; Pydantic validates every record on construction — load-bearing in a data-quality project, not a convenience.
- **Alternatives:** pip + venv + requirements.txt (no lockfile discipline); Poetry (heavier, slower); dataclasses/attrs for models (no validation).

## D-009 · 2026-07-16 · Models validate shape, not sense

- **Context:** where should bad data be rejected? Feeds deliberately contain defects (missing cost basis, mistyped ISINs, implausible dates) that the reconciliation and data-quality layers exist to catch.
- **Choice:** the canonical model enforces *shape* only — types, formats, required fields (e.g. an ISIN must look like an ISIN). Business plausibility stays representable: checksum validity is a helper method (`has_valid_checksum`), cost basis is optional, no cross-field date rules.
- **Why:** rejecting defective data at parse time destroys the evidence the platform's whole value chain (detect → route → resolve → audit) is built on. The boundary guarantees integrity of *representation*; downstream layers judge *quality*.
- **Alternatives:** strict parse-time validation (simpler, but the wrong layer owns the rules); no validation anywhere (silent corruption).

## D-010 · 2026-07-16 · Wire formats as spec-shaped subsets, XSD validation deferred

- **Context:** full ISO 20022 messages are enormous (hundreds of optional elements, deep nesting); our model carries a focused field set. Somewhere between "toy XML" and "schema-perfect" a line must be drawn.
- **Choice:** renderers/parsers use the real message structure and element names for the fields we carry (e.g. `SctiesBalCtdyRpt`, `BalForAcct/FinInstrmId/ISIN`, `AcctBaseCcyAmts/HldgVal/Amt`), with documented simplifications (flattened nesting where the spec stacks identical wrappers). Validation against official XSD schemas is a recorded backlog item, not silently skipped.
- **Why:** the learning and the parsing work are genuine at subset fidelity; schema-perfect output would consume days on optional elements nothing downstream reads. Honesty is preserved by documenting the line (module docstrings) rather than pretending.
- **Alternatives:** full schema fidelity + XSD validation now (high cost, low marginal value); invented XML tags (would falsify the "real formats" premise).

## D-011 · 2026-07-16 · Raw landing layout and generation policy

- **Context:** the generator turns from library into data factory; the layout and corruption policy shape everything downstream.
- **Choice:** (a) Hive-style `date=YYYY-MM-DD/` directories so Spark partition-prunes naturally; (b) business days only; (c) the two holdings renditions (semt.002, MT535) are corrupted **independently** — the same day's views genuinely disagree sometimes, which is what cross-feed reconciliation exists to catch; (d) everything derives deterministically from the calendar date, so any historical day regenerates byte-identically; (e) ground-truth manifests are written **outside** the raw landing directory — the pipeline must never read them; only detection-quality evaluation may.
- **Why:** each choice removes a future argument: pruning comes free, weekends don't fake volume, cross-format breaks exist by construction, investigations can reproduce any file exactly, and the DQ layer can't accidentally cheat.
- **Alternatives:** flat directories (no pruning); same corruption for both holdings formats (reconciliation would only ever catch format-coverage gaps); random non-reproducible corruption (untestable).

## D-012 · 2026-07-17 · CI authenticates to Databricks with a personal access token

- **Context:** the scheduled delivery job must authenticate to Databricks with no human present. Local dev uses a cached OAuth browser login (user-to-machine), which by definition needs a person. Machine auth offers two routes: a **personal access token** (PAT) tied to my user, or **OAuth machine-to-machine** against a **service principal** (client id + secret).
- **Choice:** a PAT, stored as a GitHub Actions repository secret alongside the workspace host. Verified end to end before committing: token creation is enabled on Free Edition, and `make land` uploads successfully with `DATABRICKS_AUTH_TYPE=pat` forced (so a cached OAuth session could not mask a failure).
- **Why:** it works on Free Edition today, needs one secret and no second identity to manage, and keeps the CI upload command byte-identical to the local one. The tradeoff is real and worth stating plainly: a PAT is a long-lived bearer credential carrying **my user's full permissions**, which is strictly weaker than a scoped service principal. It is acceptable *here* because the workspace holds nothing but synthetic data, and it is the only credential in play.
- **Mitigations:** create it with a bounded lifetime (~90 days) and rotate; GitHub masks secret values in run logs; a scheduled run failing on expiry is loud (GitHub emails on failed scheduled workflows), which is the right failure mode — better a broken job than an immortal credential.
- **Alternatives:** *OAuth M2M service principal* — the correct answer once anything real is at stake: a non-human identity, scoped, revocable without disturbing my own login. Rejected for now because Free Edition's service-principal support is unverified and it buys nothing at this scale; this is the documented upgrade path. *OIDC workload-identity federation* — no stored secret at all, the ideal end state; Databricks supports it for service principals on some tiers, unverified here and moot until a service principal exists.

## D-013 · 2026-07-17 · Bronze runs on a file-arrival trigger, defined as a bundle

- **Context:** landing files was automated (D-006), but parsing them into bronze still required a human to open the notebook and run it — half a pipeline. Automating it raises two separate questions: *what starts the job*, and *how the job is defined*.
- **Choice (what starts it):** a **file-arrival trigger** watching `/Volumes/workspace/parvum/landing/raw/`, with `wait_after_last_change_seconds: 60` and `min_time_between_triggers_seconds: 300` — not a cron.
- **Why:** a timer would have to *guess* the gap between the feed landing and the job starting, and guessing wrong is silent — the job runs, finds nothing, reports success, and the data is simply a day late. The trigger follows the data instead of a clock, so the fetch schedule can move without anything downstream needing to know. The `wait_after_last_change` window coalesces a three-file delivery into one run rather than three; `min_time_between_triggers` is a floor on frequency, protecting a free-tier compute quota from a pathological upload loop. Verified by real firing before being committed to, not merely accepted by the API.
- **Choice (how it's defined):** a **Databricks Asset Bundle** (`databricks.yml`) deployed with `make deploy-job`, with the job pulling code via `git_source` from `main` rather than from bundle-synced files.
- **Why:** a job clicked together in the Workflows UI is invisible to review, unreproducible, and absent from git — the same objection as un-applied Terraform in D-005. As for `git_source`: the notebook imports the repo's own parsers through a relative path (`../ingest/src`), so it needs the whole repo tree, not just `spark/`; a checkout gives it exactly the layout it already expects. And because `main` is branch-protected and CI-gated, "the job runs `main`" *already has* a review gate in front of it — a separate code-deploy step would add ceremony without adding control. File sync is switched off (`sync: paths: []`) so the deploy carries the job definition alone, leaving no third copy of the code to wonder about.
- **Alternatives:** *Quartz cron ~07:00 UTC* (simple and predictable, but couples to the fetch's timing and fails silently when that assumption breaks); *bundle-synced files* (the canonical bundle pattern, and the right answer if the job's code ever diverges from the repo layout — but here it would deploy a third copy of code that git already versions and CI already gates); *the UI* (rejected: not code); *Lakeflow/DLT declarative pipelines* (a heavier abstraction than a single idempotent notebook needs today).
- **Note:** the trigger fires on arrival, so it will not reprocess the 65 days already in the volume — history was loaded by the runs that preceded it, and the registry anti-join makes that distinction moot anyway.

## D-014 · 2026-07-17 · 13F seed is committed, and identifiers are derived only where the rule is real

- **Context:** D-002 committed to seeding portfolios from real SEC 13F filings. Two questions had to be settled before any of it could be used: *when* the data is fetched, and *what identifier* the securities carry — 13F reports CUSIPs, while custodial feeds carry ISINs.
- **Choice (when):** the parsed extract is **committed to the repo** (`ingest/seed/holdings_13f.json`) and refreshed by an occasional, reviewed `make fetch-13f`. Generation never touches the network.
- **Why:** the generator must produce byte-identical files for a given date (D-011), which is impossible if its input depends on a network call that can change or fail. This also makes a filing change arrive as a reviewable diff rather than a silent shift in every downstream file. The extract records the **accession number** and no retrieval timestamp: the accession pins the exact immutable filing, while a timestamp would make the file differ on every fetch and turn the scheduled check (PR 13) into a generator of meaningless pull requests.
- **Choice (identifiers):** derive the ISIN for US-domestic CUSIPs; **exclude CINS codes and record the exclusion**, with an explicit `country` parameter that callers must supply rather than a hidden assumption.
- **Why:** deriving a North American ISIN is applying ISO 6166's own arithmetic — country code + 9-character CUSIP + check digit — not guessing a mapping, so it doesn't violate D-004's ban on invented reference data. The rule stops precisely where knowledge stops: a CINS (letter-leading, e.g. Chubb's `H1467J104`) belongs to a foreign issuer whose real ISIN is `CH0044328745`, obtainable only by lookup; `US` + CINS would mint an identifier that exists nowhere, which is worse than a gap because it looks correct. The same trap is why `country` is a parameter: numeric CUSIPs are issued to Canadian issuers too, and the code cannot tell you the domicile. Phase 2's OpenFIGI mapping is what brings the excluded names back.
- **Alternatives:** *fetch at generation time* (kills reproducibility); *carry CUSIPs through the feeds and defer all mapping to Phase 2* (arguably the most faithful — real feeds do carry mixed schemes — but it would mean teaching all three renderers a second identifier scheme, and ISINs are what a custodian would actually send); *derive ISINs for everything by assuming US* (fabricates identifiers for foreign issuers — rejected outright); *hardcode a CUSIP→ISIN lookup table* (invented reference data by another name).
- **Note:** this PR reads 13F but does **not** yet swap the seed book — doing so changes every historical file, and bronze's idempotency key is `file_path`, so it would not notice the content changed. That reprocess is handled deliberately in its own step rather than smuggled in here.

## D-015 · 2026-07-17 · The seed book keeps 13F's weights and divides its scale

- **Context:** the 13F extract carries institutional quantities — 227,917,808 Apple shares, a $263bn book. A wealth-management client does not hold that. The book had to become *real* without becoming *absurd*.
- **Choice:** divide every share count by a single constant (10,000), keeping prices as filed. The result is ~28 positions and a **~$25.2M account with Apple at 23.0%** — Berkshire's genuine concentration at a plausible private-client scale. Prices come from `value / shares` (the real quarter-end price); cost basis, which 13F does not report, is synthesized deterministically from each CUSIP and left absent for ~20% of positions.
- **Why:** the *shape* of a portfolio — how many names, how concentrated, which sectors — is what makes it look like a real book rather than noise (D-002), and scaling shares uniformly preserves every relative weight exactly. The scale is honest fiction and documented as such.
- **Why a share divisor rather than scaling by value**, which sounds more principled: NVR trades near $6,590, so its 11,112 shares are $73M of the filing yet round to *zero* on any value-based scale that keeps Apple's 227.9M shares sensible. Share counts and position values are not proportional, and the data says so.
- **Alternatives:** keep the filing's raw quantities (real, but a private client holding 227M Apple shares undermines the whole fixture); scale by value (rounds small positions out of existence); fractional share quantities (avoids the rounding but reads oddly on a custody statement). A zero-share position now raises rather than being silently dropped, so a future filer that doesn't suit the divisor forces a conscious decision.
- **Note:** duplicate `security_name`s are deliberate — Alphabet, Lennar and Liberty Live are each held in two share classes: one issuer name across two distinct ISINs. Anything downstream keying on name rather than identifier should break here rather than in production.

## D-016 · 2026-07-17 · Bronze detects restatements by content, not by path

- **Context:** swapping the seed book rewrites all 195 raw files with new content **at the same paths**. Bronze's idempotency was an anti-join on `file_path` — "have I seen this path?" — which silently assumes a file's bytes never change once written. Under that check the swap would be invisible: the job would recognise every path, skip every file, report success, and leave bronze permanently disagreeing with the volume. No error, just two systems quietly diverging.
- **Choice:** compare each file against the `sha256` the registry has stored since PR 8 and never read. Path unknown → new. Digest differs → **restated**: delete that file's rows from all four tables, then re-parse. Digest matches → skip.
- **Why:** real custodians restate constantly — a corrected file for a past date, same name — so content-keyed idempotency isn't a workaround for our seed change, it's what the pipeline needed anyway. The seed swap simply became the first restatement it ever saw, handled by the mechanism rather than by a human dropping tables. Deletion precedes re-parsing (an append alone would duplicate every row) and the registry row is deleted with the rest, so a crash mid-supersede leaves the file looking un-ingested and the next run redoes it cleanly — the same ordering argument that already puts the registry write last.
- **Cost, stated plainly:** every file is hashed on every run. At hundreds of small files that is nothing; at millions you would track a source-side version or modification time rather than reading the data to discover it changed.
- **Limitation, stated plainly:** landing overwrites the raw file, so the superseded *bytes* are already gone by the time bronze notices. A platform that must prove what it received on a given day would land restatements to a versioned path and keep both. Bronze remains "raw as received" only for the *current* version of each file.
- **Alternatives:** drop the four tables and rebuild (~90s — simple and legitimate for a synthetic universe, since bronze is a rebuildable projection of the volume; rejected because it is a human fixing by hand what the pipeline should handle, and the next restatement would need the same intervention); keep both versions with an `is_current` flag or a version column (proper SCD-2 history, but pointless while the volume itself keeps only one copy of the raw file).

## D-017 · 2026-07-17 · 13F filings are pipeline input, selected point-in-time — the committed seed is gone

- **Context:** D-014 committed a parsed 13F extract to the repo. Two things were wrong with that once examined. First, reference data in git means every refresh is a source-code change — PRs proposing data, which is the wrong medium (that friction is what surfaced the issue). Second, one frozen extract meant a refresh changed the book for *every* date, past included — a mass restatement per quarter, when real books evolve forward only.
- **Choice (where the data lives):** a local **filing store** outside git (`data/edgar/`, gitignored; synced by `make fetch-13f`; fetched fresh per run in CI, which needs `SEC_USER_AGENT` as a repository secret). One directory per filing, keyed by accession number; sync is incremental because filings are immutable — an existing directory is never re-fetched, and a half-written one can't be mistaken for complete (written to a `.partial` name, renamed atomically).
- **Choice (how it's read):** **point-in-time** — the statement for `as_of` is built from the latest filing with `filing_date <= as_of`. *Filed*, not period end: holdings become knowable when the filing lands (~45 days after quarter end), and a custodian statement can only reflect what exists. This is the as-of/filed-at (bitemporal) distinction in its simplest usable form.
- **Why determinism survives leaving git:** D-014 argued the seed had to be committed for byte-identical regeneration (D-011). The property actually rests on something stronger — EDGAR filings are immutable and pinned by accession, so "the set of filings public by date D" is a fixed historical fact. Same store state, same bytes; and a new filing changes only dates at and after its own filing date. History stops being rewritable *by construction*, which retires the quarterly mass restatement (D-016 remains for genuine corrections).
- **Verified live:** with four real filings cached, the generated pile now steps at 2026-05-15 — Berkshire's Q4-2025 book (37 securities after CINS exclusions; Amazon, Diageo, Domino's…) until 2026-05-14, the Q1-2026 book (28) from the filing date on. Bronze reprocessed exactly the pre-boundary days as restatements and skipped the rest as unchanged.
- **Limitations, stated:** amendments (`13F-HR/A`) are still skipped — a faithful store would supersede the original from the amendment's own filing date (backlog); dates before the earliest cached filing refuse loudly rather than guess.
- **Alternatives:** keep the committed extract (rejected: data as source code, mass restatements); fetch at generation time with no store (network-dependent generation, kills replay); land filings only in the lakehouse volume (right for the platform's own reference data — that leg is planned — but the generator also needs local, offline access).

## D-018 · 2026-07-17 · File-arrival triggers don't see overwrites; restatement runs are chased explicitly

- **Context:** after landing 195 *overwritten* files (the PR-12 seed swap), the file-arrival trigger never fired — while a control probe at a *new* path fired in 86 seconds the same morning, and PR-10's original probe in 118. The trigger is alive; it watches for paths appearing, not bytes changing.
- **Finding:** the trigger has exactly the blind spot D-016 removed from bronze — path-based change detection — one layer up. The same bug class can hide at several layers of a pipeline, and fixing it at one layer says nothing about the others.
- **Choice:** accept and document, with an explicit operational rule: normal daily flow needs nothing (a new `date=` directory is a new path — fires; new EDGAR accessions likewise), but **any re-land of existing paths is followed by `make run-job`**. The job is idempotent and content-keyed, so the chase is always safe.
- **Alternatives:** a periodic cron alongside the trigger as a safety net (adds scheduled runs that usually no-op; may be adopted if manual chasing ever misses); landing restatements to versioned paths so every delivery is a new path (also fixes D-016's "superseded bytes are gone" limitation — the recorded upgrade if restatements become routine).

## D-019 · 2026-07-17 · The account universe: five accounts, opaque ids, one consolidated cash file

- **Context:** one account rendered three ways was enough to prove the pipeline, but it isn't what a custodian sends. A custodian services many accounts for a recipient and knows nothing about the recipient's clients; feeds are account-scoped, and the client → entity → account mapping is the wealth manager's own reference data.
- **Choice (the universe):** five accounts across three real 13F filers — Berkshire Hathaway (×2, different scales), Gates Foundation Trust (×2), Pershing Square (×1) — giving five genuinely distinct books: different sizes (~$1.6M to ~$25M), concentrations (Pershing's 11 names vs Berkshire's 28), and one **EUR-based account**, so multi-currency enters bronze now rather than being retrofitted when Phase 4's FX conversion needs it. Client grouping (3 clients) is deliberately *absent* from the custodian-side registry — it arrives as WM-side reference data in Phase 2.
- **Choice (identifiers):** account ids are **opaque custodian-style numbers** (`60011234`, `FQ5521`, `X4478210`…). Custodians issue identifiers, not descriptions; "what is account FQ5521?" is precisely the question reference data exists to answer, and losing that mapping is how real feed onboarding starts.
- **Choice (file shapes — the formats' own asymmetry):** semt.002 and MT535 are per-safekeeping-account messages → one file per account per day. camt.053 explicitly supports repeated `Stmt` blocks → **one consolidated cash file per day** carrying every account's statement. The daily delivery is 11 files (5×2 holdings + 1 cash). Parsers and bronze now handle one-file-many-statements — an assumption real feeds break, learned on a format designed for it.
- **Corruption:** independent per (account, format) — cross-account and cross-format disagreements both exist by construction (extends D-011).
- **Alternatives:** readable account ids like `ACC-GROWTH-001` (self-documenting, but hides the mapping problem feeds actually pose); per-account cash files (uniform, but would keep "one file = one statement" baked in); synthetic variations of one filer (cheaper, but the books would be visible siblings and Phase 2 would meet fewer real identifier edge cases).

## D-020 · 2026-07-17 · Issuer domiciles: a curated slice, because CUSIPs don't reveal countries

- **Context:** adding real filers made the identifier trap fire in production data: a CUSIP does not encode its issuer's domicile. CINS codes announce themselves (leading letter), but **Canadian issuers get ordinary numeric CUSIPs** — and our new filers' *top* holdings are exactly that: Canadian National Railway (Gates) and Brookfield (Pershing). Deriving their ISINs with the default "US" would mint `US1363751029` — plausible-looking, checksum-valid, and existing nowhere (the real ISIN is `CA1363751027`).
- **Choice:** a hand-curated `ISSUER_DOMICILES` map (CUSIP → country) for the non-US cross-listed names our filers hold, pinned in tests against the issuers' real published ISINs; everything else defaults to US. ADRs deliberately need no entry — a depositary receipt is a genuine US instrument whatever the underlying company's home (Diageo, InBev, Coca-Cola FEMSA).
- **Why this doesn't violate D-004:** it is a *small slice of real facts*, not invented data — the same rule that admits real ISINs admits real domiciles. The default-to-US for unlisted CUSIPs is the recorded risk; the mitigation is an audit at fetch time over every cached filing, flagging what falls through.
- **The audit already paid for itself:** it caught **Waste Connections** (`94106B101`, redomiciled to Ontario in 2016) which the initial map had missed — one character away from the genuinely-US Waste **Management** (`94106L109`). Both are pinned in a test.
- **Alternatives:** derive US for everything (fabricates identifiers — rejected outright); exclude everything not provably US (would drop the filers' largest holdings); build the securities master now (Phase 2's job — OpenFIGI + SEC/GLEIF replace this map wholesale).

## D-021 · 2026-07-17 · Ownership modelled as a validated graph; reference data stays in the ingest package for now

- **Context:** Phase 2 opens with the reference layer. The first question a wealth platform answers that the custodial feeds cannot is *who owns account FQ5521?* — feeds carry opaque account numbers and no client, entity, or legal structure. That mapping is the wealth manager's own data. Real client structures observed in the target product: transactions are owned by *legal entities* (trusts, foundations, LLCs), not people; reports roll up by "Top Level Legal Entity"; ownership views show a *percentage* per node.
- **Choice (shape):** model ownership as a validated **client → legal-entity → account DAG** with percentage edges, and resolve *effective ownership* on demand (product of edge percentages along a path, summed across paths) rather than storing a flat account → client column. Invariants enforced at construction: known endpoints, acyclic, every owned node closes at exactly 100%, every universe account reachable.
- **Why a graph:** real ownership is layered and shared — a family holds accounts through several entities, and an investment LLC is jointly owned by two families. A flat column cannot express "60% of this account's assets are ultimately the Reyes family's"; the graph makes that a traversal, and the effective-ownership calculation is the analogue of the platform's visual ownership map — the thing silver needs to attribute a position to its owners.
- **Why resolve on demand, not materialise:** the graph is the source of truth and the computation is trivial at this scale. Pre-computing effective percentages would be a cache to invalidate for no benefit; if it ever needed to scale, materialising into a table is the recorded upgrade.
- **Choice (home):** the ownership model lives in the **`parvum_ingest` package** (`parvum_ingest.ownership`), not the empty `reference/` directory, even though it is reference data. It binds tightly to the account registry (`accounts.UNIVERSE`, already in that package) and validates against it. `reference/` is reserved for the **securities master** (OpenFIGI + SEC + GLEIF — external APIs, own dependencies, licence concerns), which is genuinely separable and arrives next. That is the moment to establish `reference/` as its own package and move the reference-flavoured modules (ownership, domicile, accounts) together — splitting a ~180-line module into its own package with a uv workspace now would be ceremony ahead of need.
- **Alternatives:** flat account → client mapping (cannot express layered/shared ownership — the whole point); a graph database (correct at enterprise scale, absurd at five accounts); building `reference/` as its own package immediately (premature split; the securities master is the natural trigger); putting ownership in the canonical `model.py` (that module is feed vocabulary — ownership is WM reference data that feeds never carry, a deliberate separation).

## D-022 · 2026-07-17 · Securities master via OpenFIGI, with the unmapped kept as a visible "Unknown" bucket

- **Context:** silver needs a *what-is-this-instrument* layer to complement the ownership *whose-is-it* layer. Feeds carry ISINs — proprietary-ish, issuer-scoped; joining positions across accounts and formats, and grouping them by sector/type in gold, wants a single canonical instrument key.
- **Choice:** map the universe's ISINs to **FIGI** (Bloomberg's open instrument id) plus name/type/sector via the free OpenFIGI v3 API, producing a securities master keyed by ISIN. Every identifier OpenFIGI cannot map becomes an explicit **`mapped: false` entry — the "Unknown" bucket** — never a dropped row.
- **Why FIGI:** it's open, free, and join-friendly, exactly what a normalisation key should be; the ISIN→FIGI hop is the real reference-data work and the metadata is what gold groups by. **Why the Unknown bucket is first-class:** a security the master can't identify still sits in a client's account and must stay visible, flagged, and routable to whoever curates reference data — the target product surfaces an "Unknown" asset class rather than hiding it, and dropping the row would be silent data loss (the exact failure D-009 guards against, one layer up). A miss is data.

Had that derivation fabricated `US…` ISINs, OpenFIGI would have failed to map them and dropped them into Unknown — so a clean map is independent confirmation the domicile derivation is right.
- **Storage:** the built master is a gitignored file (`data/reference/securities_master.json`, `make build-master`), fetched occasionally and reviewed — same posture as the 13F store (D-017): reference data is pipeline input, not source code, and its stability comes from FIGIs being immutable, not from committing it. Landing it into the Unity Catalog volume for silver to read is the next slice.
- **Alternatives:** commit the master to git (reviewable, but reference-data-as-source-code, the pattern D-017 retired); a paid/bulk identifier source (unjustified at 76 instruments); drop unmapped ISINs (silent loss — the thing the Unknown bucket exists to prevent); map on FIGI's SEDOL/CUSIP endpoints too (unneeded — feeds carry ISINs).

## D-023 · 2026-07-18 · Silver: one grain per table, full rebuild, format preference over reconciliation

- **Two tables, not one.** `silver_positions` keeps one row per (date, account, security); the owner attribution is a separate `silver_position_owners` (plus the tiny materialised `silver_account_owners` bridge). The rejected alternatives: exploding owners into the positions table makes a naive SUM double-count every shared account; nesting owners as an array makes the owner roll-ups — the whole point of ownership — require an explode step in every query. One clean grain per table means the safe query is the obvious one.
- **Full rebuild every run.** Silver is a pure projection of bronze × reference, so the notebook `CREATE OR REPLACE`s its tables from scratch: idempotent and restatement-proof by construction, no incremental bookkeeping. At this scale (thousands of rows) the tradeoff is free; the incremental path (MERGE on changed dates) is the documented scale-up.
- **Bronze's two holdings copies → deterministic preference, not reconciliation.** Every position arrives in both semt.002 and MT535; silver keeps one row per grain by preferring semt.002 (the richer message), file path as tie-break. Whether the copies *agree* — which is where the seeded defects live — is a data-quality slice of its own, deliberately not smuggled into the conforming join. (Live probe while building: 14,930 bronze rows → 7,629 grains, 328 of them single-copy — the defects visibly splitting pairs, exactly the surface that later slice will measure.)
- **Silver runs as a downstream task of the bronze job**, same file-arrival trigger, `depends_on: bronze` — no second trigger to go dark (D-018), and the job's alerting (failure email, duration warning) covers the whole chain unchanged.
- **The master is data; the graph is code.** The securities master reaches the workspace by being landed into the volume (`make land-master` — no egress, D-006); the ownership graph arrives with the git checkout like the parsers do. Overwriting the master's path does not fire the trigger (D-018) — deliberate: a master refresh alone shouldn't rerun bronze; it takes effect on the next feed arrival or an explicit `make run-job`.

## D-024 · 2026-07-18 · Silver cash: separate task, native currency, and visible collapse of feed duplicates

- **Separate notebook + job task (`silver_cash`), not an extension of the positions notebook.** The positions notebook is live and proven; the two share only the ownership bridge (each builds it from code). Both depend on bronze alone and may run in parallel. Renaming or extending live, working pieces for tidiness is how "the job points at a notebook that moved" failures happen.
- **Native currency, no FX.** FQ5521's EUR stays EUR. Conversion needs a real rates source (D-004 forbids inventing one); silver states per-currency facts. Trigger to revisit: gold's cross-family totals.
- **Feed duplicates are collapsed, visibly.** Probing the real bronze before writing the CTAS found the assumed grain violated: 80 duplicate (as_of, account, transaction_id) groups — the seeded DUPLICATE_TRANSACTION defect. A transaction reference is unique by definition, so collapsing IS the conformance; hiding the collapse would starve the quality layer. Hence `source_row_count` (>1 = collapsed) and `source_disagrees` (TRUE when the copies conflicted). The probe also caught a defect *collision* nobody designed for: 6 of the 80 pairs differ on settlement_date by one day — DUPLICATE_TRANSACTION and SETTLEMENT_SHIFT landing on the same movement. Where copies conflict, the pick is deterministic (earliest settlement date, the conservative claim) and flagged rather than silently resolved.
- Same D-023 shape otherwise: one grain per table (`silver_cash_balances`, `silver_cash_transactions`), owner-attributed variants with proration done once (`silver_cash_balance_owners`, `silver_cash_transaction_owners`), full rebuild, column comments reapplied per rebuild.

## D-025 · 2026-07-18 · Reconciliation: findings as rows, dual integrity verdicts, and comparing only what both sides say

- **`dq_holdings_recon` is a long findings table** — one row per disagreement (presence gaps + null-safe field mismatches), not a wide per-position scorecard: findings are countable, groupable by class, and empty means clean. **`cost_basis` is deliberately not compared**: semt.002 structurally never carries it (0 of 7,465 rows), so the comparison would manufacture thousands of fake findings. Stated consequence: MISSING_COST_BASIS defects are invisible to cross-format recon on this feed pair — a check that would catch them (day-over-day continuity) is future work, recorded. Knowing which defects a check *cannot* see is part of the check.
- **`dq_cash_integrity` renders two verdicts per account-day** — the invariant (opening + movements = closing) evaluated against raw bronze *and* conformed silver. The pair is the point: raw-breaks/conformed-passes days prove the duplicate collapse right (63 of them); conformed breaks isolate genuinely missing movements (76). One verdict would conflate the feed's errors with ours.
- **Movement amounts are unsigned with direction in the type** (camt.053's CdtDbtInd convention, kept by the parser). The first integrity draft summed raw amounts and "found" 100% of account-days broken — a 100% failure rate indicts the check, not the data. Follow-on conformance fix in the same slice: `silver_cash_transactions` now carries `signed_amount` (and the owner proration uses it), so downstream sums stop re-deriving sign rules. The debit set mirrors `DEBIT_TYPES` in the camt renderer — the one authority on which types debit.
- **The defect manifests are grading material, not platform input.** The recon must catch defects without the answer key; the manifests (ground truth of every injection) are used at verification time to prove the catches complete. This slice's proof: 84 duplicates injected, 80 caught — and the manifests show exactly 4 DROPPED_TRANSACTION injections removed one copy of a duplicated pair, cancelling both defects. The same 4 events reconcile the collapse count (84−4=80) and the conformed break count (80−4=76), and raw breaks tie out to 84+80−21 both-defect days−4 cancels = 139. Every number closed.

## D-026 · 2026-07-18 · FX via ECB reference rates: store what was published, interpret at consumption

- Gold must add USD and EUR wealth; inventing a rate violates D-004. The **ECB's euro foreign exchange reference rates** are the cheapest real source that exists — free, keyless, stable URL, full history — and reference rates are exactly the right instrument for valuation reporting (indicative, not tradeable quotes). We slice the USD column only, from 2026 on: one real series, small on purpose.
- **Full-history file over the rolling 90-day one**: the pile window is a config knob (DAYS), and a source that silently stops covering the earliest days when someone raises the knob is a trap. History removes the edge for one ~700 KB download.
- **The store holds exactly what the ECB published** — TARGET business days only, gaps included. Calendar-completion (a rate for every day) is the *consumer's* interpretation, done by `fill_forward` at read time, tested offline, and transparent: it returns each day's rate *with the publication date it came from*, so a Saturday valuation can say "Friday's rate" instead of pretending. Refuses to start before the series does — valuing at nothing must be loud.
- **Fetched and landed by the daily workflow, non-fatally**: an ECB outage must not block the feed delivery, and gold degrades gracefully (carry-forward from the last landed file). The rates land before the feeds so the day's job run sees the day's rate. Same D-018 note as the master: overwriting the rates path fires no trigger, by design.
- The `parvum-fetch-fx` CLI lives in the **reference** package — first script there — because it touches nothing of ingest's; a CLI belongs to the package whose data it maintains.

## D-027 · 2026-07-18 · Gold: four report tables, USD headlines with labelled conversion, quality as a column

- **The four views a wealth report actually opens with** (user-chosen scope): `gold_client_wealth` (client × day headline), `gold_asset_allocation` (client × day × class, cash as its own 'Cash' class and 'Unknown' kept visible per D-022), `gold_income` (client × month × type, DIVIDEND and INTEREST only — fees and trades are flows, not income), `gold_top_holdings` (client × rank, top 10 on the latest day, one row per security summed across the client's accounts, weight on the conventional positions-value basis).

A currency the notebook cannot convert aborts the run rather than passing through at par: 1:1 is the one conversion nobody ever notices being wrong.
- **Quality is a column, not a caveat**: `books_reconcile` on the wealth table is `every(conformed_consistent)` over the client's accounts for that day, straight from `dq_cash_integrity`. A report number and the verdict on its inputs travel together.
- **Gold only sums and shapes.** All proration happened once in silver (D-023/D-024); the FX fill is the only computation gold adds, and it is imported from the tested reference package, not re-derived in SQL.

## D-028 · 2026-07-18 · Serving tenancy: schema-per-tenant, two fictional advisory firms

- **Context:** the serving layer should demonstrate multi-tenancy the way real wealth platforms do it, but the data universe has three client families and no firms. Schema-per-tenant proves nothing with one tenant.
- **Choice:** tenant = advisory firm. Two fictional firms partition the families — **Aldergate Wealth Management** (Hartwell) and **Stonefield Family Office** (Okafor + Reyes) — each with its own Postgres schema (`tenant_aldergate`, `tenant_stonefield`) carrying an identical layout, plus a data-free `tenant_template` schema that exists solely for jOOQ code generation. The same Flyway migration set is applied programmatically to every schema at startup; tenant ids are validated against `[a-z][a-z0-9_]*` because a schema name cannot be a bound JDBC parameter.
- **Why:** firm-level tenancy mirrors the real model (the platform's customer is the advisor, not the family); isolation becomes structural — a query rendered against Stonefield's schema *cannot* see Hartwell — rather than a WHERE clause every query must remember. Stonefield holding both owners of the shared 60/40 account keeps the interesting cross-owner views inside one tenant.
- **Alternatives:** tenant = client family (three schemas, but conflates firm with client and splits the shared account's owners across tenants); row-level tenancy with a tenant_id column (one schema, but isolation by discipline instead of structure — the thing this phase is supposed to demonstrate the alternative to).

## D-029 · 2026-07-18 · Gold reaches Postgres via a Python exporter; projection is truncate-and-reload

- **Context:** Databricks Free Edition cannot open connections to arbitrary hosts (D-006), so the pipeline cannot push to Postgres. Something with open egress must move gold.
- **Choice:** a Python exporter CLI (new `export/` workspace package) pulls the gold tables over the SQL Statements API — the already-proven access path — and loads Postgres: from GitHub Actions in production, against the docker-compose Postgres locally. Each load **truncates and reloads** each tenant's projection tables in one transaction. The Postgres DDL is owned by the Flyway migrations in `serving/`; because those are plain SQL files, the exporter's tests apply the very same files to a test database — one source of truth, exercised from both sides.
- **Why:** pull-from-Actions is D-006's pattern reused, with zero new infrastructure; truncate-and-reload matches gold's restatement-proof full-rebuild semantics exactly (an upsert that never deletes would silently keep rows a gold rebuild removed) and is trivially idempotent at this volume. The serving app stays read-only — its availability never depends on Databricks being reachable.
- **Alternatives:** the Quarkus app importing on startup (mixes ETL into the serving path, bloats the memory footprint D-003 cares about); file handoff via the landing volume (an extra hop that buys nothing at hundreds of rows); MERGE/upsert (accumulates, but risks drift against a full-rebuild source of truth).

## D-030 · 2026-07-18 · jOOQ code generation from the migration files (offline), and one class set across every tenant schema

- **Context:** the endpoints need jOOQ's typed classes, which are generated from a schema. jOOQ can read that schema from a live database or from the DDL directly. Separately, schema-per-tenant (D-028) means the same tables exist in `tenant_aldergate`, `tenant_stonefield`, … — but generating one class set per schema would be absurd.
- **Choice:** generate with jOOQ's **`DDLDatabase`**, which parses the Flyway migration scripts in `serving/` directly — no database runs at build time. Generated sources land in `target/` (regenerated every build, never committed, exactly as Quarkus treats its own generated code). At runtime the `DSLContext` is configured `renderSchema=false`, so every table renders as a bare name; which tenant's copy that resolves to is decided by a per-request `SET LOCAL search_path` (see the endpoints below). One generated class set therefore serves all tenants.
- **Why:** codegen-from-DDL keeps the build hermetic and fast (CI's `serving` job needs no Postgres, and `mvn generate-sources` works with Docker cold), and it makes the migration the *single* schema source of truth — codegen and Flyway read the identical DDL, the same principle as a committed lockfile. `renderSchema=false` + `search_path` is what turns structural tenant isolation into a one-line routing concern instead of a schema qualifier threaded through every query.
- **Cost paid:** `DDLDatabase` interprets the DDL in an in-memory H2, where PostgreSQL `text` becomes a non-indexable CLOB, so the projection's string columns are declared `varchar` (unbounded) — the identical type in PostgreSQL, chosen so the primary keys build under the interpreter. The accommodation is a one-line comment in the migration, not a second source of truth.
- **Alternatives:** generate against a live/testcontainers Postgres migrated by Flyway (the "most faithful" route, but adds a database to every build for a schema jOOQ can already read from the files — and the app's own tests already prove the DDL runs on real Postgres); commit the generated sources (drifts from the DDL silently, and reviewers diff machine output); one codegen run per tenant schema (multiplies identical classes to no benefit); route tenants by fully-qualifying the schema in each query (defeats the point of generating once).

## D-031 · 2026-07-18 · The ownership graph is a structural gold projection, not a monetary one

- **Context:** the serving API should let a firm see *who owns which accounts* — including the signature shared account split 60/40 between two clients — but the four existing gold tables carry only prorated money, not the account→owner edges. The edges live in `silver_account_owners`.
- **Choice:** add a fifth gold table, `gold_ownership`, that projects those edges as-is: one row per (account, owning client) with the effective fraction, an `owner_count`, and an `is_shared` flag (derived with a window over the account). It carries **no money** — the monetary attribution is already prorated into wealth/allocation/holdings. It flows to Postgres and the `/tenants/{id}/ownership` endpoint through the same exporter and schema-per-tenant machinery as the rest.
- **Why:** the ownership graph answers a structural question ("who owns what, and what is shared"), so mixing account values back in would duplicate — differently rounded — numbers the other endpoints already own. Keeping it edges-only makes it the natural place to *see* why proration exists at all. Because each edge routes to its owning client's tenant, a shared account whose owners sit in one firm (Stonefield holds both) shows both edges; one split across firms shows `is_shared` true on each side even though the co-owner is not visible — structural isolation stated honestly rather than papered over.
- **Alternatives:** enrich the graph with each account's current USD value (re-derives money the wealth endpoint already reports, at a second rounding); serve it straight from `silver_account_owners` (breaks D-029's rule that only gold reaches Postgres, and skips the derived shared-account columns); a dedicated reference/graph store like a property graph DB (real overkill for a five-account universe, and it would sit outside the one lakehouse-of-record).

## D-032 · 2026-07-18 · The frontend is a static SPA that consumes the API; dev proxies instead of CORS

- **Context:** the serving layer's read endpoints needed a face — the one part of the project a non-engineer sees. The API is a separate Quarkus service, so the question is what the frontend *is* relative to it.
- **Choice:** a **static single-page app** (Vite + React + TypeScript, charts with Recharts), built to plain files with no server of its own. It calls the API over `fetch`. In development, Vite **proxies** `/tenants` to the local Quarkus app, so the browser only ever calls its own origin and CORS never enters the picture; in production the app is served behind the same origin as the API (or pointed elsewhere by a build-time `VITE_API_BASE`). Charts use the project's validated data-viz palette, always with a legend and direct labels, and the UI is theme-aware.
- **Why:** the data is already shaped by the API — the frontend's whole job is presentation, so a static SPA is the honest amount of machinery (no SSR, no backend-for-frontend, nothing to run or scale but a CDN). The dev proxy keeps this PR free of a CORS configuration on the serving app: cross-origin is a deployment concern, decided when the two are actually hosted apart, not a thing to hardcode now. ## D-033 · 2026-07-19 · AWS CLI auth uses a dedicated IAM user via `aws login`, not static keys or Identity Center

- **Context:** the AWS deploy (D-005) needs the CLI and Terraform authenticated against a real account. AWS offers three paths: root user keys, a static IAM-user access key, or IAM Identity Center (SSO) with short-lived tokens. Identity Center was tried first for its auto-expiring credentials, but enabling it prompted creating an AWS Organization — which immediately converts this account from the Free Plan to pay-as-you-go pricing and **forfeits the free-tier credits**. That trade is wrong for a personal account whose whole point is spending the credit deliberately.
- **Choice:** a dedicated IAM user (`parvum-terraform`, not root) with console access enabled, using AWS's newer **`aws login`** flow (managed by the `SignInLocalDevelopmentAccess` policy) instead of a permanent access key: it authenticates via a browser against the user's console password and issues temporary credentials that auto-rotate every 15 minutes and expire within the session (up to 12h). Because Terraform's AWS SDK doesn't understand that `login_session` credential type, a second CLI profile (`parvum-tf`) wraps it via `credential_process = aws configure export-credentials --profile parvum --format process` — a supported SDK-wide mechanism that re-emits the session as plain temporary keys on demand.
- **Why:** gets the safety property that motivated Identity Center (nothing permanent on disk, short-lived credentials) without its Organization prerequisite or the credit loss. `AdministratorAccess` is attached directly rather than a scoped policy — Terraform needs to create IAM roles for App Runner/RDS, and hand-scoping further buys little on a single-owner account where the same person already controls everything root can do.
- **Gotcha recorded:** `credential_process` failed silently (`exit status 1`, no visible stderr) when the command path was quoted with spaces (`"C:/Program Files/Amazon/AWSCLIV2/aws.exe" ...`) — Terraform's invocation doesn't handle that quoting the way a shell does. Fixed by pointing at the Windows short (8.3) path instead: `C:\PROGRA~1\Amazon\AWSCLIV2\aws.exe`. Also: the S3 **backend** block resolves credentials independently of the `provider "aws"` block, so `profile = "parvum-tf"` had to be set in both places, not inherited.
- **Alternatives:** static IAM access key (simplest, but a permanent secret sitting in `~/.aws/credentials` indefinitely); IAM Identity Center/SSO (the more standard answer at organizational scale, rejected here solely because enabling it on this account forfeits the free-tier credit); root user access keys (never — root is the account's superuser and AWS explicitly recommends against generating keys for it).

## D-034 · 2026-07-19 · Terraform state on S3 with native locking; a budget alert is the first resource created

- **Context:** the AWS deploy needed a place for Terraform state to live before any other resource could be provisioned, per the D-005 guardrail that a budget alert must exist before anything billable does.
- **Choice:** state lives in a versioned, encrypted, public-access-blocked S3 bucket (`parvum-tfstate-<account-id>`), using Terraform's native S3 locking (`use_lockfile = true`, needs Terraform ≥1.10) instead of a DynamoDB lock table. That bucket is created by a small, separate `infra/terraform/bootstrap/` config that keeps its own state **local** — the chicken-and-egg problem of "state needs a bucket; the bucket's creation needs to be tracked in state" is avoided by keeping that one piece out of the remote backend entirely. The main config's very first applied resource is an `aws_budgets_budget` (50%/80% of a $20/month threshold, email alert) reusing the same `ALERT_EMAIL` already configured for Databricks job failures.
- **Why:** native S3 locking removes what used to be a mandatory extra always-on resource (DynamoDB) for a solo project with no concurrent applies to guard against; versioning means a bad apply's state is recoverable. One alert email for both Databricks and AWS keeps "who gets paged" a single fact instead of two addresses to keep in sync.
- **Alternatives:** local-only state for the main config too (simplest, but no backup/recovery and a single point of failure on one laptop for something meant to stay up, unlike the bootstrap module's one trivial resource); DynamoDB-backed locking (the pre-1.10 standard pattern, now an unnecessary resource); skipping the budget alert (violates the D-005 guardrail directly, and the account's Free Plan hard cap doesn't communicate *why* spend crossed a threshold before it does).

## D-035 · 2026-07-19 · App Runner is replaced by ECS Express Mode (amends D-005)

- **Context:** D-005 named AWS App Runner for the Quarkus container, and its own guardrail said to verify App Runner's availability before writing Terraform. That check came back positive only in the abstract — the concrete failure showed up on the first `terraform apply`: `SubscriptionRequiredException: The AWS Access Key Id needs a subscription for the service`. Research confirmed why: **AWS closed App Runner to new customers on 2026-04-30** (existing customers keep working; it's in maintenance mode). This account, having only just started using AWS, is a new customer by that definition, so provisioning is blocked outright — not a quota or config issue.
- **Choice:** **ECS Express Mode** (`aws_ecs_express_gateway_service`, added to the AWS Terraform provider in v6.23.0 — this project moved off the `~> 5.0` constraint to get it). It is AWS's own direct successor, announced for exactly this transition: point it at a container image, get a public HTTPS endpoint, with its own managed Application Load Balancer, ACM certificate, and auto scaling — the infrastructure role (`AmazonECSInfrastructureRoleforExpressGatewayServices`, AWS-managed) provisions and owns all of it. Tasks run in the default VPC's public subnets (Express Mode's own default), each getting a public IP, so ECR pulls and CloudWatch logging work without a NAT gateway.
- **Why:** preserves D-005's original intent (a PaaS-like public endpoint with nothing to hand-manage) at no extra charge over the underlying Fargate compute, and keeps the "no NAT gateway, no ALB to operate ourselves" guardrail intact — Express Mode's ALB is AWS-managed, not a resource this project owns or pays a separate fixed rate for.
- **Alternatives:** full ECS Fargate + a self-managed Application Load Balancer (the literal fallback D-005 named; more Terraform, more moving parts, and a small fixed ALB cost, worth revisiting only if Express Mode — a very new feature — turns out to have rough edges that matter here); a third-party PaaS (Render, Fly.io, Railway — moves the project off AWS entirely, contradicting D-005's "one cloud" reasoning).
- **Known rough edge:** `terraform plan` on `aws_ecs_express_gateway_service` shows a phantom diff on `primary_container.environment` values and computed fields (`ingress_paths`, `service_revision_arn`) even immediately after a clean apply — confirmed cosmetic by checking the container's actual boot logs (the real deployed value was correct both times). Matches the resource's own CHANGELOG, which already lists a similar "inconsistent result after apply" fix for `network_configuration.security_groups` — this is a very new resource type (November 2025) with rough edges, not a bug in this config.

## D-036 · 2026-07-19 · RDS is publicly accessible, protected by TLS + password (amends this session's original private-VPC design)

- **Context:** RDS was first built private (VPC-only, reachable only from the app's security group) as the safer default. But the exporter (`export/`, D-029) already runs `export-gold` from wherever has open egress to both Databricks and Postgres — originally GitHub Actions' hosted runners, per its own code comment written before this session's AWS work began. Hosted runners are on the open internet with unpredictable, unallowlistable IPs; they cannot reach a private-VPC-only database, and giving them access via a NAT gateway would add exactly the fixed monthly cost D-005 ruled out.
- **Choice:** `publicly_accessible = true` on the RDS instance, with the security group open on 5432 to any source, defended instead by **`rds.force_ssl = 1`** (a DB parameter group forcing every connection to negotiate TLS) and the existing Terraform-generated 32-character password. The Quarkus datasource JDBC URL carries `?sslmode=require` so the app itself refuses a plaintext connection too.
- **Why:** network-perimeter isolation (the original design) and IP-allowlisting both assume a caller with a stable, knowable address — true for the ECS service, not true for GitHub Actions. ## D-041 · 2026-07-19 · silver_positions dedupes by winning *file*, not winning *row* — a mistyped identifier was double-counting positions

- **Context:** found while validating the upcoming performance slice (TWR/Dietz/IRR) against live gold data before writing any Spark SQL — the daily wealth series had unexplained single-day spikes that fully reverted the next day (Hartwell: +12.2% on 2026-07-01, back down the day after), which would have made any return computed on top of it look broken or manufactured. The notebook's own prior comment ("semt.002 preferred over MT535 ... file path as a final deterministic tie-break") already described this whole-file intent; the implementation had keyed the preference at the wrong granularity (per matching row, not per file), which is what let a corrupted key slip through as if it were a legitimately-missing counterpart.

The corrupted row still lands under the wrong ISIN in silver (a real, correctly-flagged data-quality problem — `dq_holdings_recon` catches it as a `MISSING_IN_MT535`/`MISSING_IN_SEMT002` pair, unchanged by this fix), but it can never again inflate the dollar total.

Every prior day's small ±0.1–12% wiggle was entirely this bug; none of it was real.
- **Alternatives:** keep the row-level dedupe but add a post-hoc "same market_value appearing under two different identifiers same account-day" detector to null one out (a patch on the symptom, not the cause — and fragile, since a legitimately different security could coincidentally share a value); switch the tie-break to prefer whichever copy's identifier passes ISIN checksum validation (fixes this one defect type but not any future defect that corrupts identifiers a different way — the whole-file choice generalizes to every defect in the holdings pool, not just this one).

## D-043 · 2026-07-19 · DQ metrics are a declarative rollup over the existing detail tables, not a new source of truth

- **Context:** the next item in the roadmap's sequencing after the performance slice: a queryable DQ metrics table (freshness/completeness/accuracy/exception counts) plus an ops KPI dashboard. This item had a specific trigger already waiting in the code: D-040's writeup explicitly promised "the planned day-over-day continuity DQ check" once the cash book had real continuity to check against — that check had never been built.
- **Choice:** two new tables in `dq_recon.py`. `dq_cash_continuity` (grain: account × date) is a new *detail* table answering a question the existing `dq_cash_integrity` doesn't: not "does today's own arithmetic add up" (intra-day) but "does today's opening equal yesterday's closing" (inter-day) — computed via `LAG(closing) OVER (PARTITION BY account_id ORDER BY as_of)` against `silver_cash_balances`, `NULL` on each account's first date per the same boundary convention D-042 already established for performance. `dq_metrics` (grain: date × dimension × metric) is a *rollup*: one `UNION ALL` branch per named check, aggregating `dq_holdings_recon`, `dq_cash_integrity`, `dq_cash_continuity`, and `bronze_file_registry` into a single consistent shape — freshness (one row per rebuild, dated at the run date, since staleness is a fact about *now* not about a historical day), completeness (files-landed rate), accuracy (three rates: cross-format match, intra-day cash arithmetic, day-over-day continuity), and exceptions (the raw counts behind those three rates, for trend/aging charts a rate alone would hide).
- **Why:** "declarative" means adding a future check costs one more `SELECT` in the `UNION ALL`, never a schema migration — the whole point of a rollup table over a growing set of independently-evolving detail tables. Rollup, not replacement: every number in `dq_metrics` still traces to a detail-table row with full lineage, matching this project's standing "findings are rows, not logs" principle rather than collapsing detail into an opaque score. `passed` is populated only where a threshold honestly applies (rate metrics against a target); exception counts stay `NULL` there deliberately — a count is trend data for a chart, forcing a pass/fail verdict onto it would manufacture a false signal.
- **An honest number, not a green dashboard:** accuracy rates sit around 40–100% day to day in this fixture, because defects are injected at a known, deliberate rate (D-011) — a KPI dashboard reporting "accuracy SLA attainment: ~60%" here is the *correct* reading for a fixture built to exercise detection, not a bug to chase to 100%. Verified live before writing the notebook: 65/65 business days have all 11 expected files (completeness is a clean 100%, as expected — the defect pool never drops a whole file); the day-over-day continuity check reports 0 breaks against the *clean* `silver_cash_balances` chain (D-040 already guarantees that), confirming the new check is correctly wired before it ever sees a corrupted delivery to catch.
- **Alternatives:** a wide table (one row per date, one column per metric) instead of the long/declarative shape — rejected because every new check would then be a schema migration, defeating the stated goal; computing freshness per historical `as_of` to match the other three dimensions' grain — rejected because "was 2026-05-01 fresh" isn't a coherent retrospective question the way "was it complete/accurate" is; a day that arrived late shows up as a *completeness* gap on its own date, not a freshness fact about a day in the past. A single freshness row per rebuild, dated at the rebuild's own run date, is the only framing that is actually true.

## D-046 · 2026-07-20 · A separate internal app, with real (if minimal) auth — reopening D-044's reasoning

- **Context:** Phase 6 (alts HITL) needs a human review queue with write actions and an audit trail — a genuinely different audience and risk profile from the public, read-only client dashboard. D-044 explicitly declined a second app for the Ops page for exactly the opposite reason: no auth need, no justification for the extra infrastructure. That reasoning cuts the other way once a real access-control boundary is actually needed, so this decision revisits it rather than silently overriding it.
- **Choice, three parts:**
  1. **A new app, `internal/`** — same Vite/React/TS/Vitest toolchain as `web/`, its own Vercel project, calling the same Quarkus API under a new `/internal/**` path prefix. No new backend service: the API is reused, not duplicated (`parvum.tenants`/`TenantSchemas` untouched).
  2. **Auth is one shared credential, not a user table.** `AuthResource` compares a submitted password to a single server-held secret via constant-time equality (`MessageDigest.isEqual`), the same trust model the RDS password already uses (D-036: an SSM `SecureString`, resolved only at container start, rotated by updating the parameter). Bcrypt/Argon2 exist to defend a stored table of *many* hashes against a database leak; there is no such table here, so hashing one comparison value buys nothing a constant-time compare doesn't already give, and skips a new dependency.
  3. **Sessions are a stateless signed cookie** (`SessionToken`: HMAC-SHA256 over an expiry timestamp, verified with `MessageDigest.isEqual`), not a session table — no database round-trip to check a request is authenticated, and nothing to clean up on expiry. The cost: logout can only tell the browser to drop the cookie, not revoke a token server-side; a captured token stays valid until its 12-hour expiry regardless. Acceptable for one internal credential behind a private URL, revisit (a session table, or short-lived tokens + refresh) if this ever needs real multi-user revocation.
- **CSRF, cheaply:** `InternalAuthFilter` requires a custom header (`X-Parvum-Internal`) on every `/internal/**` request, including login/logout. A cross-site form submission cannot set a custom header; only same-origin `fetch` calls from the internal app's own code can (`internal/src/api.ts`) — a standard mitigation that costs one header check, chosen over a CSRF-token handshake as disproportionate for a single-credential internal tool.
- **Why now, not later:** building the auth boundary first (this slice) before anything sits behind it means every subsequent HITL/Ops endpoint is authenticated from the moment it exists, rather than retrofitted — the exact gap D-044 flagged as the reason *not* to build a second app prematurely no longer applies once there's a real write-action surface coming.
- **Verified:** `mvn verify` green (16 tests, incl. `AuthResourceTest` covering the missing-header/wrong-password/valid-session/logout paths); full manual run against the live dev server and through `internal/`'s own Vite proxy — 403 (no header) → 401 (no cookie) → 401 (wrong password) → 204 (correct password, real `Set-Cookie` captured) → 204 (session check with that cookie) → 204 (logout, empty cookie returned).
- **Alternatives:** Clerk or Auth0 (hosted, managed auth) — considered and set aside for this single-shared-credential use case; still the natural upgrade path if the internal app ever needs real per-user accounts or SSO. Folding HITL into `web/` as a third unauthenticated tab (D-044's own precedent) — rejected because it would make write/approve actions on financial-document extractions publicly reachable, undercutting the audit-trail story that is the point of Phase 6.

## D-047 · 2026-07-20 · The alts document generator: a small deterministic fund waterfall, four defect types, real PDFs

- **Context:** Phase 6's first real building block — synthetic capital-call/distribution/capital-account documents to eventually extract, validate, and route to human review. Unlike the custodial feeds (Phase 1), there's no open wire-format spec to build against; the shape has to be designed from how these documents actually work.
- **Choice, four parts:**
  1. **A tiny fund universe (2 funds) tied to the existing account universe.** `FUND-PE01`/`FUND-VC01` each roll up to an existing custody account (`X4478210`, `FQ5521` from `parvum_reference.accounts.UNIVERSE`) rather than inventing a parallel client universe — a real LP's fund interests sit alongside their brokerage accounts on one relationship, and this keeps that true here too.
  2. **A deterministic waterfall (`book.py`), not a calibrated PE/VC model.** Four calls (60% of commitment, unevenly spaced), two distributions (return of capital, then a capital gain), ten quarterly capital account statements — every statement's `ending_balance` reconciles exactly against its own beginning balance and that period's flows *by construction*, and chains into the next statement's beginning balance. No randomness in the clean book at all (a fixed, hand-written unrealized-gain-percent walk, not an RNG) — the only randomness anywhere in this generator is *whether* a defect fires, in `generate.py`, so the clean book a corrupted document is compared against during eventual validation is never itself ambiguous.
  3. **Four defect types**, all semantic (corrupt the model before rendering, so the PDF stays well-formed — the same split `parvum_ingest.defects` established): `MISSING_FIELD`, `COMMITMENT_MISMATCH`, `AMOUNT_TRANSPOSITION` (call/distribution notices), `ARITHMETIC_ERROR` (capital account statements, breaking the reconciliation invariant #2 relies on). `AMOUNT_TRANSPOSITION` swaps the *leading* two digits, not the trailing ones — this fixture's amounts are round to the nearest hundred/thousand, so a trailing-digit swap is usually a silent no-op (caught by a failing test before this shipped, not assumed).
  4. **Real PDFs via reportlab** (`render.py`), not styled HTML or a text file with a `.pdf` extension — a later extraction step should do genuine document-reading work, the same reason the custodial feeds are real ISO 20022/SWIFT rather than a convenient JSON shape.
- **Why:** this is deliberately a smaller, more mechanical slice than the custodial feed generator — two funds, sixteen documents each, one defect per category — because the interesting engineering in Phase 6 is downstream (extraction confidence, deterministic validation, the review queue's audit trail), not in the synthetic-data generator itself. Keeping the fund book intentionally simple (no J-curve calibration, no real vintage benchmarks) is a conscious scope cut, not an oversight.
- **Verified:** 23/23 tests green (deterministic book construction, every statement's reconciliation invariant, defect injection recorded with real targets, PDF text extraction confirming the real figures — including a corrupted one — are actually present in the rendered document). Real end-to-end run, not just tests: generated the full 32-document set, read a real corrupted PDF back with `pypdf`, and confirmed both a `COMMITMENT_MISMATCH` ("Cumulative Called $1,751,000.00" where the clean book says $1,750,000.00) and an `AMOUNT_TRANSPOSITION` ("Call Amount $100,000.00" where the clean book says $1,000,000.00) are legible in the extracted text exactly as injected.
- **Alternatives:** seed the fund universe from real GP/LP data — no such open data exists for private funds, which is the entire premise for going synthetic here (see the project brief); a syntactic ("garbled/unopenable file") defect category mirroring `TRUNCATED_FILE` — deferred, since a corrupted PDF file doesn't demonstrate "realistic messiness" the way a truncated XML/text file does, it just fails to open; more defect types (e.g. a call-sequence gap) — deferred to keep this slice reviewable, additive later the same way `ingest`'s defect pool grew over multiple phases.

## D-049 · 2026-07-20 · LLM extraction: forced tool-use, hybrid confidence, ground truth is the *rendered* document not the clean book

- **Context:** the first real "AI-native" step in the pipeline — read each alts PDF back to structured fields via Claude, scored against the generator's own ground truth. `docs/ARCHITECTURE.md`'s fetch/process split already says where this has to run: Databricks Free Edition can't reach the open internet, so this is a GitHub Actions step, not a notebook.
- **Choice, four parts:**
  1. **Forced tool-use, one tool per document type**, not a free-text prompt parsed afterward. `tool_choice={"type": "tool", "name": ...}` guarantees the response is a JSON-schema-conformant object — the failure mode this avoids is a model that writes a reasonable-looking paragraph instead of the exact shape the pipeline needs, which would need its own brittle parser to recover from.
  2. **Ground truth is the *rendered*, possibly-corrupted document, not the clean book.** `generate.py`'s manifest now carries each document's full `fields` (the corrupted model's own `model_dump`), not just the injected-defect diff. Extraction's job is to read what is actually printed — a `COMMITMENT_MISMATCH`-corrupted number is genuinely printed wrong on the page, and a faithful extraction should report exactly that wrong number. Reconciling or flagging it is deterministic validation's job (a later slice), not extraction's — conflating the two would make it impossible to tell "the LLM misread a correct document" apart from "the LLM correctly read an incorrect document."
  3. **Confidence is hybrid, not purely self-reported.** The model states its own confidence (asked explicitly, honestly, in the system prompt); `self_consistency_ok` independently checks the *extracted* values for internal arithmetic/presence problems (does beginning + flows = ending; are required amounts present) and caps confidence at 0.5 if that check fails — a floor a self-reported score alone can't provide, since the model has no way to independently verify its own read. This check is single-document only (it needs nothing beyond the one extraction result), so it runs in `extract.py` itself and works in production, not just eval; cross-document checks (commitment continuity across a fund's whole call history, call-sequence gaps) need the whole fund's documents together and belong in the later Databricks validation slice.
  4. **Claude Haiku, not a larger model.** Structured extraction from a one-page document with a forced schema is exactly the kind of task a fast/cheap model handles well; there's no reasoning-depth requirement here that would justify Sonnet/Opus cost for 32 documents (or more, as the fund universe grows).
- **Eval scope, deliberately narrower than the full north-star metric:** `evaluate.py` reports document exact-match rate and field-level accuracy — a read-fidelity measure. The roadmap's north-star metric for this phase ("straight-through-processing rate at a fixed escaped-defect budget") needs a confidence threshold and an auto-accept-vs-review routing decision to be meaningful, and that routing logic doesn't exist until deterministic validation (the next slice) does. Shipping a narrower, honest metric now rather than a wider one that can't yet be computed correctly.
- **Verified — and one genuine live blocker found, not hidden:** 37/37 tests pass, all against a mocked Anthropic client (the test suite has zero API cost by design — `alts-extract`/`alts-eval` are real, billed runs, deliberately kept out of default CI, matching `export-gold`'s precedent for anything with a real-world side effect). A single real extraction call was attempted before landing this to confirm the wiring actually works end to end; it failed with `Your credit balance is too low to access the Anthropic API` — the Anthropic Console account (separate from a Claude Pro subscription, which carries no API credit) has none loaded. This is recorded honestly rather than claimed as verified: the code is believed correct from the mocked tests and the schema/response handling, but a real document has not yet been successfully extracted. Follow-up: once credits are added, rerun the single-document smoke check before trusting a full-batch `make alts-extract`.
- **Alternatives:** a free-text prompt + a regex/JSON-extraction parser on the response — rejected in favor of forced tool-use, which removes an entire class of "the model almost got the format right" failures for a marginal prompt-engineering cost; purely self-reported confidence — rejected per point 3; running extraction inside a Databricks notebook via `%pip install anthropic` — technically possible but violates the platform's own stated constraint (Free Edition's restricted egress) and would be inconsistent with why every other external call in this project runs in GitHub Actions.

## D-050 · 2026-07-20 · Cross-document validation lives in a tested package, not inline notebook code — and two real bugs it caught in its own test

- **Context:** `extract.py` (D-049) deliberately stops at single-document self-consistency; the cross-document checks it named as out of scope — commitment continuity across a fund's whole call history, call-sequence gaps, capital-account statement chaining — need this slice's silver step.
- **Choice, three parts:**
  1. **The logic lives in `parvum_alts_hitl.validate`, a plain, pytest-tested Python module — not inline in the Databricks notebook.** `spark/silver_alts_documents.py` imports it (same sys.path trick `bronze_alts_ingest.py` already uses for `naming.py`) and is orchestration only: load `bronze_alts_extractions`, group by fund, call `validate_fund_documents`, write the result. This follows the project's own standing rule (brief rule 9: data-quality logic needs a test that shows a seeded break being caught) in a way inline notebook code structurally can't — a notebook cell isn't a unit a test can import and call directly.
  2. **`bronze_alts_ingest.py` grew a second responsibility** (registering `bronze_alts_extractions` alongside the existing `bronze_alts_documents`) rather than a third notebook — both are "register what landed in the alts volume," a natural single job, now with a shared `discover`/`supersede`/`register` helper trio instead of two copies of the same restatement logic.
  3. **The output is a routing decision (`auto_accept` / `needs_review`), never a correction.** `validate.py` never edits an extracted value — fixing a flagged document is the human reviewer's job (the review queue, next slice), not this notebook's. `route()` requires self-consistency AND cross-document validity AND confidence ≥ 0.85 for `auto_accept`; any one failing sends it to review.
- **Two real bugs, caught by the same integration test, not by inspection.** Writing `test_validate_integration.py` (run `validate.py` against the *real* generator output via `generate.generate()`, simulating faithful extraction, and assert every seeded defect gets caught) surfaced two wrong assumptions in the test itself before it validated anything:
  1. **"No injected defect → never flagged" was wrong.** A running-sum check cascades: if call #1's `call_amount` is corrupted, every later call's `cumulative_called` check also stops tying out, even though calls #2–4 are individually clean on the page. That's the *correct*, honest behavior of a real running-total check — a human reconciling a fund's calls by hand would flag every subsequent cumulative too, not just the first bad one — not a false positive to suppress.
  2. **Hardcoding `self_consistent=True` for the simulated extraction was wrong.** A faithfully-extracted statement whose own printed numbers don't reconcile (`ARITHMETIC_ERROR`) should compute `self_consistent=False` via `extract.py`'s own single-document check — that defect is caught *there*, not by `validate.py`'s chaining check, which only compares a statement's `beginning_balance` to the *prior* statement's `ending_balance` and has no reason to notice one statement's own internal arithmetic being wrong. The fix: compute `self_consistent` in the test via the real `extract.self_consistency_ok`, not a hardcoded value — which also made explicit, for the first time in code (not just in D-049's prose), exactly which defect type each of the two mechanisms is responsible for.
- **Verified:** 53/53 tests pass (16 new: unit tests per validator function demonstrating a specific seeded break each, plus the integration test against real generator output). `databricks bundle validate` confirms the updated job config (a new `silver_alts` task, chained after `bronze_alts`) is syntactically sound.

## D-051 · 2026-07-20 · The review-queue reverse-sync design, and why this slice stops at the Postgres/Quarkus boundary

- **Context:** the one genuinely open design question flagged when Phase 6 kicked off (D-046): once a human corrects a `needs_review` document, how does that correction get back into Databricks, the platform's system of record? Weighed three shapes with the user before building anything:
  1. **A land-file reverse-sync** — mirrors the existing fetch/land contract exactly, reversed: a periodic job reads decided-but-unsynced rows from Postgres and writes them to a new landing path; Databricks merges them into silver on its next run. Keeps Delta as the single system of record (Postgres stays a disposable projection everywhere, the stated principle in `docs/ARCHITECTURE.md`), at the cost of one more moving part and non-instant latency.
  2. **Direct write from Quarkus to the Databricks SQL Statements API on approve/correct** — no landing file, no separate job, corrections land immediately. Rejected: it would give the live Java service a new Databricks credential to manage, and open a second, less auditable write path into the lakehouse outside the normal medallion flow — a real regression from the fetch/process split every other external call in this project follows.
  3. **Let Postgres be authoritative for just this one table** (skip syncing back at all) — rejected outright: it breaks the "every Postgres table is a disposable projection, rebuildable from gold at any time" invariant this project states explicitly, for the sake of skipping one job.
  - **Chosen: option 1**, confirmed with the user rather than assumed.
- **This slice (Postgres schema + Quarkus backend) ships first; the reverse-sync itself is a follow-up commit, not bundled in.** Building the queue mechanics (load, list, approve, correct, audit) and the sync job in one PR would make either piece harder to review on its own, and the sync job's design depends on nothing this slice doesn't already provide (a `synced_at` column, reserved but unused until the sync job exists to set it). Small reviewable steps over one large one.
- **Schema choice: a new non-tenant `internal` Postgres schema, migrated separately.** The review queue is firm-ops data — no advisory firm owns a review decision — so it doesn't belong in `tenant_*` (client-facing) the way every other serving table does. A new `InternalSchema` bean (mirrors `TenantSchemas`, migrating exactly one schema from a separate `db/migration_internal` location) and `InternalQuery` (mirrors `TenantQuery`'s `SET LOCAL search_path` mechanism, fixed to one schema since there's nothing to route between) keep this cleanly separate from the tenant migration set — the tenant DDL can never accidentally pick up a review-queue table, and vice versa. jOOQ codegen gained a second `<execution>` in `pom.xml` (its Maven plugin supports per-execution `<configuration>`, not just one shared block) targeting `db/migration_internal` into a separate `dev.parvum.serving.jooq.internal` package.
- **JSONB for extracted/decided fields, not a fixed wide table** — same reasoning as `bronze_alts_extractions.fields_json`: the field set differs by `doc_type`, and forcing three doc types' worth of columns into one table would mean most rows are mostly NULL.
- **No `decided_by` column.** This app has one shared credential, not per-user accounts (D-046) — there is no real identity to attribute a decision to. Recording a constant string would manufacture accountability the system doesn't actually have; the column is simply absent rather than filled with a placeholder, and its addition is named as the natural trigger for ever needing real per-user auth here.
- **A decided item can't be decided again** (`approve`/`correct` on a non-`pending` row → 409). Keeps the audit trail meaningful as a record of the *first* human decision; re-opening a decided document for a second look is a distinct, not-yet-built workflow, not silently allowed through the same endpoint.
- **Verified live, not just by test:** `mvn verify` (26/26, Testcontainers Postgres) plus a real run against the local docker-compose database — login → list → correct → confirmed `decidedFields` and `decidedAt` set correctly → re-decision attempt correctly rejected with 409. Migration confirmed applying cleanly (`Migrating schema "internal" to version "1 - alts review queue"`) in both paths.
- **Alternatives:** a `decided_by` column defaulting to a fixed string — rejected per above; reusing the tenant Flyway/jOOQ machinery for this table instead of a second schema/codegen execution — rejected because the review queue isn't tenant data and forcing it through the tenant path would either duplicate it per tenant (D-044's tradeoff, wrong here since there's real write traffic to keep consistent) or require `TenantSchemas` to special-case one non-tenant schema, muddying a mechanism that's clean today.

## D-052 · 2026-07-20 · Two LLM providers behind one interface — Anthropic direct, and OpenRouter as the default

- **Context:** the Anthropic Console account hit a zero credit balance (D-049) and repeated card-add attempts failed. Rather than block the whole extraction slice on one payment processor, the user asked for a second provider (OpenRouter — a different billing account, and a unified gateway that can still route to Claude) with the plumbing kept flexible: cheaper/default provider for routine extraction, Anthropic direct available as an explicit escalation for a genuinely harder document.
- **Choice:** an `LLMProvider` abstract base (one method, `extract(tool, document_text) -> dict`) with two implementations. `AnthropicProvider` wraps the `anthropic` SDK's native Messages/tool-use API, unchanged from D-049. `OpenRouterProvider` wraps the `openai` SDK pointed at `https://openrouter.ai/api/v1` (OpenRouter speaks the OpenAI-compatible chat-completions/function-calling shape, not Anthropic's native one) and translates the *same* tool schema (`name`/`description`/`input_schema`) into OpenAI's `{"type": "function", "function": {...}}` envelope, parsing the response's JSON-*string* arguments (`json.loads`) instead of Anthropic's already-parsed dict. `extract_document()`, `self_consistency_ok()`, and everything in `validate.py` are unchanged — they only ever see the provider-agnostic extracted-fields dict, never which provider or model produced it (beyond a `model` string carried through for lineage).
- **Default provider is now `openrouter`, not `anthropic`** — `PARVUM_LLM_PROVIDER` (env) / `--provider` (CLI), both defaulting to `openrouter`; `anthropic` is one flag/env-var away for a harder document. This is a genuine behavior change from D-049, made explicitly at the user's direction, not a quiet default flip.
- **Both providers point at a Claude Haiku-class model today** (`DEFAULT_MODELS`) — OpenRouter's value is the billing/account independence and the option to route to a *different* provider's model later, not a forced change of model quality right now. Per-document-complexity routing (e.g. auto-escalating a low-confidence extraction to a bigger model) is a natural next step but not built here — this slice provides the plumbing (a provider is just a constructor argument), not a routing policy nothing has asked for yet.
- **Verified:** 59/59 tests pass (17 new/changed) — `AnthropicProvider`/`OpenRouterProvider.extract()` each tested against a mocked SDK client (proving the request-shape/response-parsing translation), `extract_document()` tested against a small fake `LLMProvider` (decoupling the hybrid-confidence logic from either SDK's shape), `build_provider()` tested for defaults/overrides/unknown-name. No real API call in the test suite — both real integrations remain to be smoke-tested once a working key exists for either provider (D-049's Anthropic blocker is unresolved; OpenRouter is untested live pending a key).
- **A real SDK-version snag, fixed before it shipped:** the installed `openai` SDK validates credential presence at client *construction*, not just at call time — building an `OpenRouterProvider` with no key configured raised immediately, breaking a test that only wanted to check the default model string. Fixed by resolving to a harmless placeholder string when no real key is configured, so building the object never requires a credential; an actually-wrong or missing key still fails loudly at the real API call, which is the correct place for that failure, not at object construction.
- **Alternatives:** LiteLLM or a similar multi-provider abstraction library — rejected as unnecessary weight for two providers with a shared interface this small; making `OpenRouterProvider` always request a specific underlying model rather than accepting any OpenRouter-routable string — rejected, since the whole point of a gateway is not hardcoding one provider's namespace into this codebase.

## D-053 · 2026-07-20 · The Phase 6 pipeline verified live, end to end — and a real schema bug it found

- **Context:** with a working OpenRouter key, the first real extraction call succeeded — but the very first *batch* run (all 32 documents) showed every single capital-account statement coming back `self_consistent=False`, far above the ~25% `ARITHMETIC_ERROR` injection rate. That's the signature of a systematic bug, not real defects, and it was chased down rather than shrugged off as "the model being imprecise."
- **The bug:** only `_CALL_TOOL`'s `call_amount` field carried the instruction "no currency symbol or thousands separators" in its JSON-schema `description`. Every other amount field, across all three tools, had none — so the model (reasonably, un-instructed) formatted them the natural way a person would write a dollar figure: `"$750,000.00"`. `Decimal(str(value))` cannot parse that, so every statement's `beginning_balance`/`ending_balance`/etc. parsed to `None`, and `self_consistency_ok` correctly reported "can't verify" for all of them — not a model failure, a schema-authoring gap.
- **Fix, two parts:**
  1. A single `_AMOUNT_DESC` constant applied to *every* monetary field in all three tool schemas, not just one — the instruction is now structurally impossible to forget on a new field, since there's one place it's written.
  2. **Defense in depth**, not just a prompt fix: a new shared `parvum_alts_hitl.parsing.parse_decimal()` strips a stray `$`/comma before parsing, used by `extract.py`, `validate.py`, and `evaluate.py` alike (previously three independent, slightly-differently-behaved copies of `Decimal(str(value))`). A model that mostly-but-not-perfectly follows a formatting instruction is a realistic failure mode this project's own "realistic messiness" premise should expect, not something to assume away twice more elsewhere in the codebase.
- **Full pipeline verified live, for the first time, after the fix:** `make alts-extract` (32/32 real documents via OpenRouter/`anthropic/claude-haiku-4.5`) → `make alts-eval`: **100% document exact-match rate, 100% field accuracy** against the generator's own ground truth — every printed value, including the deliberately corrupted ones, read faithfully. A high needs_review rate on calls is the correct signature of that cascade, not a sign the checker is miscalibrated.
- **Verified:** 59/59 tests still green after the refactor (parsing logic centralized, behavior preserved). This entry's live numbers are the first end-to-end real-data confirmation of D-046 through D-052 — every prior "not yet verified live" note in this phase is now closed.

## D-054 · 2026-07-22 · The review-queue export-side loader — upsert-and-flag, not truncate-and-reload

- **Context:** D-051 shipped the review queue's Postgres schema and Quarkus backend but deliberately stopped there — nothing yet moved a `needs_review` document from Databricks into it. This slice is that loader: the second leg of D-051's three (loader → reviewed-in-place → reverse-sync), following the same land-file-reversed design already settled for the sync job.
- **Why this can't reuse `export_gold.py`'s truncate-and-reload:** every gold table is a pure, stateless projection — reloading it can always start from an empty table because nothing downstream has written to it. `alts_review_queue` is different: once loaded, a human can approve or correct a row, and that decision (`status`, `decided_fields`, `decided_at`) is real state a later load must never destroy. The loader is therefore an upsert keyed on `(fund_id, document)`, gated by `WHERE status = 'pending'` on the `ON CONFLICT DO UPDATE` — a decided row is untouched no matter what the fresh fetch reports, by construction, not by a runtime check.
- **What happens to a pending row that drops out of the fresh needs_review set** (silver got rebuilt with a corrected extraction, or cross-document validation no longer flags it): weighed two options with the user before building — delete it (mirrors gold's philosophy: the queue should only ever reflect what currently needs review), or flag it and leave it in place. **Chosen: flag it** — a new `stale` boolean column (`V2__alts_review_queue_stale_flag.sql`), set `true` by the loader when a pending document is no longer in the fresh fetch, reset to `false` automatically if the document reappears. A reviewer may already be looking at a row; silently deleting it out from under them is worse than surfacing "this may no longer need review" and letting a human decide. `AltsReviewResource.QueueItem` gained the field so the (still-unbuilt) frontend has it to render whenever it lands.
- **The join this loader depends on:** `silver_alts_documents` (D-050) carries the routing decision but not the extracted values — those live in `bronze_alts_extractions.fields_json`. `review_queue_source.py` joins the two over the SQL Statements API (`WHERE routing = 'needs_review'`), the same pull pattern `gold_source.py` already established, with `confidence` cast to `DECIMAL` in the query itself so the existing typed-value converter handles it without a new `DOUBLE` case added just for this one column.
- **`_resolve_token` extracted into a shared `databricks_auth.py`** — the second CLI needing the identical DATABRICKS_TOKEN-or-mint-from-CLI logic was the trigger to stop duplicating it; `export_gold.py` now imports it too.
- **Verified:** 29/29 export tests pass (11 new — a pure-Python suite for the join row-shaping, plus a real-Postgres suite against a throwaway schema migrated with the real `migration_internal` DDL, covering: fresh load, pending-row refresh, decided-row immunity to both a refreshed and an absent fresh fetch, stale-flagging, un-staling on reappearance, and JSONB/date/null round-trips). `mvn verify` 26/26 (`AltsReviewResourceTest`'s 9 unaffected by the added column). `make lint`/`make fmt` clean across all four Python packages.
- **Not yet built:** the frontend to actually review anything (still nothing renders the queue), and the reverse-sync job itself (D-051's third leg) — both remain the natural next slices.
- **Alternatives:** truncate-and-reload like gold, accepting that a reload mid-review would silently wipe a pending decision — rejected outright once framed that way, not seriously considered; a `reviewed_at`/optimistic-lock column instead of the `WHERE status = 'pending'` upsert guard — rejected as more machinery for the same guarantee the guard already gives for free.

## D-055 · 2026-07-22 · The review-decision reverse-sync — the loop closes, additively

- **Context:** D-051's third and last deferred piece. A human can now approve or correct a `needs_review` document (D-054's loader put it in front of them), but that decision lived only in Postgres — a disposable projection, per this project's stated invariant, which meant a decided document's corrected values had no path back to Delta, the platform's actual system of record. This slice is that path, completing the land-file design D-051 already settled and confirmed with the user rather than reopening it.
- **Shape, mirroring the forward loader in reverse:** `review_decision_source.fetch_unsynced_decisions()` reads every `approved`/`corrected` row in `alts_review_queue` with `synced_at IS NULL`. `review_decision_sync.write_decision_files()` writes one JSON file per decision to a local staging directory (`data/alts/reviewed/<fund_id>/<document>.decision.json`), and `mark_synced()` sets `synced_at` — but only ever *after* a real land succeeds, never optimistically, so a failed or partial `databricks fs cp` leaves every decision eligible for the next run to retry, no manual cleanup needed. The CLI (`sync_review_decisions.py`) owns all three steps in one transaction-shaped sequence, the same combined shape D-054's loader already established, rather than splitting fetch/land/mark across separate `make` targets where a human could run one and forget the others.
